# VoxCanvas 设计文档

## 1. 项目概述

VoxCanvas 是一款纯语音控制的 AI 绘图工具。用户不使用鼠标和键盘，仅通过语音指令完成画布创作，包括创建图形、调整颜色与位置、编辑已有对象、撤销操作以及执行复杂绘图任务。

本项目面向七牛云 XEngineer 活动中的「AI 语音绘图工具」选题，重点关注以下目标：

- 指令理解准确：将自然语言语音指令稳定转换为可执行绘图操作。
- 交互容错友好：当语音识别或指令理解存在歧义时，支持确认、追问与回退。
- 响应延迟可控：简单指令尽量本地解析和执行，复杂指令再调用 AI 能力拆解。
- 复杂指令可拆解：将“一句话画一个场景”拆分为多个画布操作。
- 创作过程可编辑：所有图形对象保留结构化状态，支持后续修改、删除、撤销和重做。

### 3.2 指令解析策略

系统采用“规则优先，AI 补充”的指令解析策略：

1. 简单高频指令使用本地规则解析，例如创建基础图形、改色、移动、撤销和清空。
2. 中等复杂指令使用结构化解析，将自然语言转换为统一的操作对象。
3. 复杂场景指令调用大模型进行任务拆解，要求模型输出受约束的 JSON 操作列表。
4. 所有 AI 输出都需要经过 schema 校验，非法操作会被拒绝或转入确认流程。

示例操作结构：

```json
{
  "type": "create",
  "shape": "circle",
  "color": "red",
  "position": "top-right",
  "size": "medium"
}
```

## 4. 交互流程

### 4.1 基础流程

1. 用户点击或通过浏览器授权开启麦克风。
2. 用户说出绘图指令。
3. 系统将语音转换为文本。
4. 指令解析器判断指令类型。
5. 简单指令直接生成画布操作。
6. 复杂指令进入 AI 拆解流程。
7. 画布执行操作并更新图形状态。
8. 系统显示识别文本、执行结果和必要反馈。

### 4.2 容错流程

当系统无法明确理解指令时，进入容错流程：

- 识别文本为空：提示用户重新说一遍。
- 图形类型未知：提示支持的图形类型。
- 目标对象不明确：询问用户要操作哪一个对象。
- 颜色、大小或位置缺失：使用默认值，或在关键场景下追问。
- AI 拆解结果非法：丢弃非法步骤，并提示用户换一种表达。

## 5. 系统架构设计

### 5.1 前端模块

| 模块 | 职责 |
| --- | --- |
| VoiceInput | 管理麦克风权限、语音识别状态和识别文本 |
| CommandParser | 将识别文本解析为结构化操作 |
| CommandPlanner | 对复杂指令进行拆解，可接入大模型 |
| CanvasEngine | 管理图形渲染、对象状态和布局 |
| HistoryManager | 管理撤销、重做和操作记录 |
| FeedbackPanel | 展示识别结果、执行步骤、错误提示和确认问题 |
| Exporter | 导出画布图片 |

### 5.2 数据模型

基础图形对象计划采用统一结构：

```ts
type ShapeObject = {
  id: string;
  type: "circle" | "rect" | "triangle" | "line" | "text";
  x: number;
  y: number;
  width: number;
  height: number;
  rotation: number;
  fill: string;
  stroke: string;
  text?: string;
  createdAt: number;
  updatedAt: number;
};
```

画布状态计划采用：

```ts
type CanvasState = {
  shapes: ShapeObject[];
  selectedId?: string;
  history: CanvasOperation[];
  future: CanvasOperation[];
};
```

## 6. 延迟优化策略

为了降低从语音到绘图的响应延迟，计划采用以下策略：

- 本地优先：常见指令不调用大模型，直接由规则解析器处理。
- 分层解析：先尝试命令词和实体抽取，失败后再进入 AI 拆解。
- 渐进反馈：语音识别完成后立即展示文本，执行前展示系统理解的操作。
- 操作轻量化：画布只保存结构化图形对象，不依赖生成式图片作为主流程。
- 异步 AI：复杂指令调用 AI 时显示“正在拆解指令”，避免用户误以为系统卡住。

## 7. 复杂指令拆解策略

复杂指令会被拆成多个原子操作，例如：

用户指令：

```text
画一间房子，旁边有一棵树，右上角有太阳
```

可能拆解为：

```json
[
  {
    "type": "create",
    "shape": "rect",
    "color": "brown",
    "position": "center",
    "size": "large"
  },
  {
    "type": "create",
    "shape": "triangle",
    "color": "red",
    "position": "above-last",
    "size": "large"
  },
  {
    "type": "create",
    "shape": "circle",
    "color": "green",
    "position": "left",
    "size": "medium"
  },
  {
    "type": "create",
    "shape": "circle",
    "color": "yellow",
    "position": "top-right",
    "size": "medium"
  }
]
```

拆解结果不会直接无条件执行，需要经过以下检查：

- 操作类型是否在允许范围内。
- 图形类型是否受支持。
- 颜色、位置、尺寸是否可以映射到画布参数。
- 是否存在危险或无意义操作。
- 如果目标对象引用不明确，则转入确认流程。

## 8. 容错与确认设计

### 8.1 语音识别容错

- 展示识别文本，便于用户知道系统听到了什么。
- 支持常见同音和近义表达，例如“圆”“圆形”“圈”。
- 对无效指令给出简短提示，而不是静默失败。

### 8.2 指令理解容错

- 对缺省属性使用合理默认值。
- 对不明确对象进行追问。
- 对多对象匹配结果优先选择最近创建或当前选中的对象。
- 支持撤销，降低误操作成本。

### 8.3 反馈方式

计划支持文字反馈，后续可扩展为语音反馈：

- “已创建红色圆形。”
- “我找到了两个圆形，请说明要修改左边还是右边。”
- “没有识别到支持的绘图指令，请换一种说法。”

## 9. 技术方案

### 9.1 初步技术栈

| 层级 | 计划方案 |
| --- | --- |
| 前端框架 | React + TypeScript + Vite |
| 绘图渲染 | SVG 或 Canvas，优先考虑 SVG 便于对象编辑 |
| 语音识别 | Web Speech API 或浏览器语音识别能力 |
| AI 拆解 | 可选接入大模型 API，输出 JSON 操作列表 |
| 状态管理 | React state 或轻量状态库 |
| 测试 | Vitest，用于测试指令解析器和操作 reducer |
| 部署 | GitHub Pages、Vercel 或其他静态站点平台 |

### 9.2 选择 SVG 的原因

优先采用 SVG 作为画布渲染方案，原因是：

- 图形对象天然可编辑。
- 每个元素都可对应一个结构化对象。
- 便于实现选择、移动、缩放和导出。
- 对几何图形绘制类项目足够轻量。

## 13. 风险与应对

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 浏览器语音识别兼容性不一致 | 用户无法稳定输入语音 | 优先支持 Chrome，并在 README 中说明环境要求 |
| 语音识别文本不准确 | 指令解析失败或误操作 | 支持近义词、默认值、确认流程和撤销 |
| 复杂指令输出不可控 | 画布结果不符合预期 | 使用 JSON schema 校验和允许操作白名单 |
| LLM 调用延迟较高 | 复杂指令响应慢 | 简单指令本地解析，复杂指令异步反馈 |
| 功能范围过大 | 无法按时完成 | 优先 P0 能力，P1/P2 根据时间逐步实现 |
