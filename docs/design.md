# VoxCanvas 指令能力设计文档

## 1. 项目概述

VoxCanvas 是一款面向七牛云 XEngineer「AI 语音绘图工具」选题开发的纯语音控制绘图应用。用户通过浏览器麦克风说出「智能绘图，……」形式的自然语言命令，即可完成图形创建、对象编辑、复杂场景生成、历史操作、画布调整和作品导出。

本项目的设计重点是：

- 纯语音交互：除首次浏览器麦克风授权外，主要绘图流程不依赖鼠标和键盘。
- 指令理解准确：将口语化中文命令转换为受约束的结构化绘图命令。
- 容错与安全：不明确或危险的编辑命令不直接执行，必要时进入澄清流程。
- 低延迟与可控成本：简单命令本地解析，复杂语义命令再调用 AI Planner。
- 复杂指令拆解：支持将一句话中的多个操作拆成有序步骤执行。
- 可编辑场景：复杂场景不是生成图片，而是拆成带语义信息的基础 SVG 图形，便于后续继续编辑。

## 2. 总体实现方案

系统采用「本地规则解析 + AI Command Planner + 安全校验 + SVG 画布执行」的架构。

### 2.1 前端模块

| 模块 | 作用 |
| --- | --- |
| `useVoiceInput` | 封装浏览器 Web Speech API，持续监听中文语音并输出临时/最终识别文本。 |
| `wakeWord` | 识别「智能绘图」及近似唤醒词，只执行唤醒词后的命令内容。 |
| `parseCommand` | 本地解析低风险命令，例如清空、撤销、重做、导出、基础图形创建。 |
| `normalizationPolicy` | 判断命令是否需要 AI 纠偏或语义规划。 |
| `useCommandPlanner` | 调用后端 AI Planner，并保存规划状态。 |
| `commandValidator` | 校验 AI 返回的 JSON 是否符合标准命令结构和当前画布上下文。 |
| `canvasOperations` | 执行 create、move、recolor、resize、scene、batch 等命令并更新画布状态。 |
| `targetDescriptions` / `clarification` | 检测目标缺失或歧义，并支持用户追加一句话完成澄清。 |
| `DrawingCanvas` | 使用 SVG 渲染所有图形对象。 |
| `CanvasSceneSummary` | 根据当前画布对象生成持续更新的语义场景描述。 |
| `CommandFeedbackPanel` | 展示系统理解结果、执行结果和关键参数。 |

### 2.2 后端模块

后端使用 Express 提供 `/api/plan-command` 接口，负责：

- 读取本地 `.env` 中的 `DEEPSEEK_API_KEY`。
- 接收用户命令、本地解析结果、当前画布对象、语义分组和 sceneSpace。
- 构造受约束 prompt，要求 DeepSeek Chat 输出标准 JSON 命令。
- 为复杂场景提供 few-shot 示例，引导模型输出可编辑 scene graph。
- 返回模型 JSON，由前端再次校验后执行。

API Key 不进入浏览器端，也不应提交到 Git 仓库。

## 3. 指令设计原则

### 3.1 命令格式

系统内部统一使用结构化命令表示用户意图，主要 action 包括：

```ts
type CommandAction =
  | "create"
  | "move"
  | "recolor"
  | "resize"
  | "delete"
  | "align"
  | "arrange"
  | "resizeCanvas"
  | "scene"
  | "addSceneObject"
  | "batch"
  | "undo"
  | "redo"
  | "clear"
  | "export"
  | "unknown";
```

简单命令可以直接执行；复杂命令必须先通过 schema、目标匹配和安全规则校验。

### 3.2 本地解析与 AI 规划边界

| 类型 | 优先策略 | 原因 |
| --- | --- | --- |
| 清空、撤销、重做、导出 | 本地解析 | 意图明确，延迟低，不需要调用模型。 |
| 明确基础图形创建 | 本地解析 | 例如「画一个红色圆形」，规则稳定且成本低。 |
| 文本插入 | AI Planner 优先 | 需要区分文字内容和颜色/位置描述，规则容易误切。 |
| 场景生成 | AI Planner | 需要将语义对象拆成多个基础图形。 |
| 已有场景增量添加 | AI Planner | 需要理解参照物、对象语义和新增内容边界。 |
| 语义对象编辑 | AI Planner | 需要识别「房子」「屋顶」「树冠」等 group/part。 |
| 多步复杂编辑 | AI Planner | 需要保留顺序并输出 batch。 |
| 批量对齐/排列/改色 | AI Planner + 校验 | 需要根据数量、范围和当前画布对象匹配目标。 |
| 取消意图 | 本地规则 | 需要快速阻止误执行。 |

## 6. 未完成能力与原因说明

| 未完成能力 | 当前状态 | 未完成原因 | 后续方案 |
| --- | --- | --- | --- |
| 自由画笔 | 未实现 | 赛题重点是语音指令到绘图操作，当前优先完成结构化图形和场景编辑；自由画笔需要连续轨迹采样、笔刷状态和更复杂的语音控制。 | 后续可增加 brush action，支持「开始画笔」「向右画一条弧线」「停止画笔」。 |
| 图层控制 | 未完整实现 | 当前对象有 zIndex 并可渲染层级，但尚未提供完整语音命令如「置顶」「下移一层」。 | 增加 layer action，支持 bringForward/sendBackward/bringToFront/sendToBack。 |
| 锁定/解锁 | 未实现 | 已有安全门禁和目标确认，但没有对象锁定状态字段，也没有阻止编辑锁定对象的执行链路。 | 在 ShapeObject 上增加 locked 字段，并在 canvasOperations 中拒绝修改 locked 对象。 |
| 语音播报反馈 | 未实现 | 当前反馈为文字展示，浏览器 TTS 需要处理打断、排队和噪声回录问题；优先保证绘图链路稳定。 | 后续使用 SpeechSynthesis API，增加可关闭的语音播报。 |
| 离线语音识别 | 未实现 | 当前 Web Speech API 依赖浏览器能力，离线模型会显著增加包体、部署复杂度和浏览器兼容成本。 | 可选接入 Whisper.cpp、Vosk 或云端 ASR，但需重新评估延迟与成本。 |
| 任意复杂语义目标识别 | 部分实现 | 系统能处理 groupLabel、partLabel、颜色、形状、位置和数量，但「靠近房子的那个圆」等视觉关系需要更完整的空间推理。 | 增加目标检索评分器，结合 bbox 距离、方向、遮挡关系和 AI 解释。 |
| 更丰富基础形状 | 部分实现 | 当前支持圆形、矩形、三角形、线条、文本；复杂物体主要由这些基础形状组合。 | 增加椭圆、多边形、星形、箭头等 shape，并扩展 validator 和 renderer。 |
| 作品云端保存 | 未实现 | 目前主要支持本地导出，未接入登录和云端存储。 | 后续可引入作品 JSON 保存、导入和分享链接。 |

## 8. 成本与延迟控制

为了控制运营成本和响应延迟，系统采用以下策略：

- 本地优先：清空、撤销、重做、导出、明确基础图形创建直接本地执行。
- AI 只处理必要场景：文本、复杂场景、复杂编辑、语义对象和纠偏才调用模型。
- 结构化上下文压缩：只向 AI 发送必要的画布对象、语义分组、bbox、允许能力和少量示例。
- 输出 JSON 白名单：限制模型只能返回受支持 action，降低二次修复成本。
- 失败不执行：AI 输出不合法时阻止执行，避免错误操作扩大损失。

## 9. 结论

最终版本已经覆盖赛题要求中的核心能力：

- 纯语音控制绘图创作。
- 指令理解与容错。
- 语音到绘图操作的反馈链路。
- 复杂指令拆解与顺序执行。
- AI scene graph 生成和可编辑语义对象。
- 设计文档、README、demo 视频和测试方式。

仍未完成的部分主要集中在自由画笔、图层控制、锁定/解锁、离线语音识别和更强视觉空间推理。这些能力需要新增状态模型、渲染能力或更复杂的目标推理机制，超出了当前比赛周期中优先保障的核心语音绘图闭环。
